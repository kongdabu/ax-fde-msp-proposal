## 1. Executive Summary (1페이지)

### 왜 (시장 변화)
- 한국 클라우드 MSP 시장은 2023년 약 7조원 → 2026년 약 12조원으로, 연 15~30% 고성장 중이다(가트너·IDC 인용). 그러나 성장 동력은 "사람을 더 투입하는 SI"가 아니라 "AI로 운영 성과를 내는 모델"로 이동 중이다.
- 전통 맨먼스(M/M, 인력 투입량 기반 과금) SI는 한계에 도달했다. SW기술자 임금이 매년 4.7~8.6% 오르고(2025년 일평균 41.5만원) 인력 확보가 어려워, 매출이 인력 수에 비례하는 구조는 마진과 확장성이 막힌다.
- 고객의 요구가 바뀌었다. "인력 몇 명"이 아니라 "장애 빨리 복구, 운영비 절감"이라는 성과를 원한다.

### 무엇을 (AX FDE)
- **고객의 기존 MSP 운영 현장에 내재화되어, AI로 운영 성과를 내고 그 성과로 값을 받는** 전담 엔지니어 조직(AX FDE)을 신설한다.
- 검증된 기술 위에 선다. AIOps 도입 기업의 74%가 MTTR(장애복구시간)을 단축했고, GenAI 적용으로 티켓 분류 시간이 44% 줄었다(Moogsoft·Workativ).

### 어떻게 (차별화 — 전 축 관통 가설)
> **"운영 내재화형 FDE" — 고객의 기존 MSP 운영에 AI를 적용하되, 특정 플랫폼에 묶지 않고, 성과로 값을 받는 모델.**

| 차별화 축 | 글로벌 FDE (Palantir형) | 한국 AX FDE (우리) | 어느 축이 구현하나 |
|----------|------------------------|-------------------|------------------|
| 목적 | 자사 플랫폼(Foundry) 확산 (product-led) | 고객 **운영 성과** 창출 (outcome-led) | ① 전략 |
| 플랫폼 | 자사 플랫폼 종속 | **플랫폼 비종속·고객환경 적응형** | ④ 툴 (OTel 표준+모델 게이트웨이) |
| 가치입증 | 제품 채택률 | **운영 KPI**(MTTR↓·자동화율↑·운영비↓) | ③ 프로세스 (게이트·정산식) |
| 출발점 | 신규 고가치 프로젝트 | **기존 MSP 운영계약 고도화**(Land-and-Expand) | ① 전략 + ② 조직 |
| 과금 | 라이선스+엔지니어 프리미엄 | **맨먼스 → 성과기반 과금 단계 전환** | ① 전략 + ③ 정산식 + ⑤ 평가보상 |
| 한계비용 | 코어팀이 표준 제품화 | **재사용 Asset로 한계비용↓, 1인 커버리지↑** | ④ 툴 + ② 운영모델 |

- 조직은 **CoE(전문가센터)+고객별 상주팀 하이브리드**다. 자산·방법론·품질·양성은 본사 CoE가 중앙에서, 실행은 현장 상주팀이 맡는다 — "현장 내재화"와 "자산 재사용"을 동시에 충족하는 유일한 형태.
- 프로세스는 **PoC→운영 전환 게이트(G1)**로 "PoC 함정"(시연만 반복, 운영 미전환)을 차단한다. 성과는 베이스라인 대비 개선분을 정산한다.
- 사람은 **내부 운영인력 재교육 중심 + 외부 핵심 시니어 채용**으로 채운다(한국 AI 인재 품귀 대응).

### 기대효과 (KPI)
- **사업성과**: AX FDE 매출, 영업이익률, 성과기반 계약 비중(목표/추정).
- **운영성과(가치증명)**: MTTR 단축률, 티켓 자동화율, 운영비 절감액, SLA 준수율.
- **선행지표**: PoC→운영 전환율, 양성 완료 FDE 수, 재사용 Asset 수, 1인당 운영 커버리지.

### 권고 (경영진 결정 요청)
- 중견~대형 MSP 기준 **초기 전담조직 약 30명**(가정), **버티컬 2~3개**(금융·제조·공공), **기존 운영고객 1~2곳 등대고객**에서 PoC→운영 전환→성과과금 전환의 단계적 입증 경로를 탄다.
- 4-Phase 로드맵: P1 기반(0~6M, 12명) → P2 파일럿(6~12M, 22명) → P3 확산(12~24M, 30명) → P4 고도화(24M~).

> *주: 회사 규모·예산·현 인력 구성이 미입력되어 30명·버티컬 2~3개·소싱 비율 등은 "가정"으로 진행한다. 실데이터 입력 시 갱신한다.*

## 2. 시장·전략·차별화

### 2.1 시장 정의 — "성장하는 MSP, 그러나 모델 전환기"
- **규모/성장**: 국내 클라우드 MSP 시장 2023년 약 7조원 → 2026년 약 12조원, 연 15% 성장 전망(가트너). 일부 전망은 향후 5년 연평균 30%. *출처 간 편차가 크므로 "약/추정" 표기, 내부 영업데이터로 재검증 권고.*
- **경쟁구도**: IDC 평가 국내 MSP 1위 삼성SDS 점유율 23.9% — 대형 SI·보안기업 진입으로 "대경쟁 시대".
- **구조변화**: 2025년 SW기술자 일평균임금 41.5만원, 전년 대비 +4.7%(2023년 +8.6%). 맨먼스 단가 상승 + 인력난 → 인력투입형 SI의 마진·확장성 한계.

### 2.2 FDE 모델 적합성 — "왜 지금, 왜 한국에 맞나"
- 한국 IT는 맨먼스 중심 → 매출이 인력 수에 비례 → 임금 상승 시 마진 압박, 인력난 시 성장 제약.
- FDE는 매출을 **인력 수가 아닌 운영 성과**에 연동 → 자동화로 1인 커버리지를 키워 임금 상승을 마진으로 흡수.
- 성과의 기술적 근거가 이미 검증됨: AIOps 도입사 74% MTTR 단축, GenAI 티켓분류 44% 단축, 63% 운영효율 개선.

### 2.3 차별화 — "운영 내재화형 FDE"
- Palantir FDE는 "서비스처럼 보이지만 실제는 제품개발 전략"이다. FDE가 만든 현장 커스텀("gravel road")을 코어팀이 표준 제품("paved highway")으로 흡수해 **Foundry 플랫폼 확산**을 노린다.
- 우리는 자사 플랫폼이 강제 자산이 아니고, 고객의 멀티벤더 운영환경에 적응해야 하므로 product-led를 복제할 수 없다.
- 차별화 = ① 플랫폼 비종속 ② 운영 KPI로 가치 입증 ③ 기존 운영계약 Land-and-Expand ④ 성과기반 과금. 단, "현장자산→재사용자산" 루프는 차용한다(④축 자산 전략).

### 2.4 경쟁 포지셔닝
> "우리는 인력을 파는 SI도, 플랫폼을 파는 CSP도 아니다. 고객 운영 현장에 들어가 **AI로 성과를 내고 성과로 값을 받는** 운영 AI 파트너다."

### 2.5 수익모델 — 맨먼스 → 성과기반 3단계 전환
| 단계 | 과금구조 | 성과정산 적용 | 전제 게이트 |
|------|---------|--------------|------------|
| 1단계 | 맨먼스(M/M) | 없음(현행) | 착수(G0) |
| 2단계 | 맨먼스 + 성과 인센티브 | 베이스라인 대비 개선분 일부를 인센티브로(소액·검증용) | PoC 통과(G1) |
| 3단계 | 성과/구독 | 정산식 본격 적용, 맨먼스 비중↓·성과 비중↑ | 지속성과(G2) |

- 재사용 Asset(④축)으로 한계비용을 낮춰 성과 과금의 마진을 구조적으로 떠받친다.

### 2.6 GTM
- 신규 영업이 아니라 **기존 MSP 운영계약의 AI 고도화(Land-and-Expand)**로 시작 — 이미 운영 데이터·신뢰·상주 인력이 있는 곳.
- 등대고객 1~2곳 PoC → 성과 베이스라인 → 운영 전환 → 레퍼런스 자산화 → 버티컬 확산.

**표준 스키마 — 전략축 핵심 Action Item (요약)**

| 항목 | 목표 | 핵심 Action Item | 담당 | 시점 | KPI |
|------|------|-----------------|------|------|-----|
| 시장 정의 | 사업 정당성 확보 | 시장 규모/성장 데이터팩, 맨먼스 한계 진단서 | 사업기획 | P1 | 시장팩·진단서 |
| 차별화 | 운영 내재화형 FDE 명제화 | Palantir 분석노트, 4차별화축 전제 공유 | 전략 | P1 | 4축 공유 |
| 수익모델 | 성과기반 매출 비중↑ | 과금 3단계 모델, 성과지표 계약화 | 전략+재무 | P1~P2 | 모델 승인 |
| GTM | 성과 레퍼런스 조기 확보 | 버티컬 2~3개 확정, 등대고객 PoC 착수 | 전략+영업 | P1~P2 | 버티컬 확정·PoC 2건 |

## 3. 조직·거버넌스

### 3.1 조직 형태 — "CoE + 고객별 상주팀" 하이브리드 (권고)
- 차별화 가설은 ① 운영 내재화(→현장 상주)와 ② 재사용 Asset로 1인 커버리지 확대(→중앙 자산화)를 **동시에** 요구한다. 아래 옵션 C만 둘을 함께 충족한다.

| 옵션 | 형태 | 적합도 |
|------|------|--------|
| A. 독립 전담조직 | 별도 사업부 분리 — 운영 단절로 "내재화" 차별화 훼손 | △ |
| B. 순수 매트릭스 | 기존 운영조직에 분산 — 책임 분산, 자산 표준화 불가 | △ |
| **C. CoE + 상주팀 (권고)** | **본사 CoE가 자산·방법론·품질·양성, 현장 상주팀이 실행** | **◎** |

- CoE는 Palantir의 "코어팀"(현장 자산을 표준 자산으로 흡수) 역할을 한다.

> 핵심 안전장치: "PoC→운영 전환"은 **사업총괄이 최종 승인(A)**, CoE 품질이 자문(C)으로 견제 — 영업 압박으로 미성숙 PoC가 운영 전환되는 것을 막는다.

### 3.5 채용·소싱 — 내부 전환 + 외부 핵심 채용
- **한국 인사 현실**: "AIOps·LLMOps + 운영(MSP)" 복합인재는 시장에 희소. 외부 채용만으로 30명을 채우는 것은 비현실적이고 단가 부담이 크다.
- 사내 MSP 운영인력은 **고객 운영환경·도메인 지식**을 이미 보유 — 운영 내재화형 FDE의 핵심 자산. AI 역량만 재교육하면 된다.
- **레벨별 소싱 차등**: L4 Lead·Solution Arch는 외부+사내 핵심 발탁 혼합 / L3~L2 FDE는 사내 운영 우수인력 전환 주력 / 횡단 AIOps/Platform은 외부 채용 비중↑ / L1은 사내 학습 트랙.

> **[정합성 검수 — 소싱 비율 상충]** ② 조직은 "내부 60 : 외부 40", ⑤ 역량은 "내부 70 : 외부 30"으로 다르다. **LEAD 권고: 전사 기준값을 "내부 65±5 : 외부 35±5"로 통일**하되, 레벨별 차등(시니어·횡단직은 외부 비중↑, L2~L1은 내부 비중↑)을 우선 적용한다. 확정값은 HR의 사내 전환 가능 인력 풀 스크리닝 결과로 P1에 정한다. (부록 A 참조)

### 3.6 운영 모델 — 상주·교대·다고객 동시지원
| 모델 | 적용 고객 | 방식 |
|------|----------|------|
| 상주형 | 금융·공공(규제·보안 민감) | 고객사 상주 (망분리·보안 충족) |
| 하이브리드(기본 권고) | 제조·중견 | 주 N일 상주 + 원격 |
| 원격+CoE 백업 | 자동화 성숙 고객 | 원격 + 본사 CoE 지원 |

- **24x7 온콜**: 공통 SRE/온콜 풀(2명+순환)로 야간·주말 다고객 동시 온콜. AIOps 자동화로 야간 호출 자체를 줄이는 것이 1차 목표.
- **1인 커버리지 확대**: 초기엔 1팀=1고객 전담으로 신뢰·성과를 만들고, Phase 3부터 횡단 지원으로 확대.
- **KPI 일원화**: 고객별 운영 KPI 대시보드(MTTR·자동화율·운영비)를 고객 보고 + 내부 평가 + 성과 정산의 공용 근거로 일원화.

## 4. 프로세스·방법론

### 4.1 전체 그림 — Engagement → Delivery 생애주기
```
[Engagement 영업]  E1 기회발굴 → E2 자격검증 → E3 제안·계약
                          ══(핸드오프 게이트 G0)══▶
[Delivery 운영]    D0 Discovery → D1 진단·베이스라인 → D2 PoC
                          ▣ 전환 게이트 G1 (PoC→운영)  ← "PoC 함정" 탈출
                   D3 운영 전환 → D4 내재화(자동화·자산화)
                          ▣ 확장 게이트 G2 (업셀·성과과금 전환)
                   D5 확장(Land-and-Expand) → 새 유스케이스로 순환 ↺
```
> **전통 SI와의 차이**: SI는 검수로 직선 종료. FDE는 G1·G2 게이트를 통과하며 운영에 남아 순환·확장한다. 매출이 "납품 완료"가 아니라 "운영 성과"에서 나온다.

### 4.2 방법론 D0~D6 (단계별 핵심)
| 단계 | 목표 | 완료기준(DoD) | 주관 역할(②) | 필요 툴(④) / 필요 역량(⑤) |
|------|------|--------------|-------------|--------------------------|
| **D0 Discovery** | 페인포인트·유스케이스 식별, 측정목표 합의 | PoC 유스케이스 1~2개+목표 KPI 서면 합의 | FDE Lead(+영업) | Observability·ITSM 추출 / 운영도메인·퍼실리테이션 |
| **D1 베이스라인** | 성과 정산 출발선 데이터 고정 | 고객이 베이스라인 수치·측정식 서명 | FDE(데이터)+고객 | Observability·BI / 데이터분석·KPI설계 |
| **D2 PoC** | 실데이터로 목표 KPI 이동 검증(섀도 우선) | G1 정량기준 충족 데이터 제시 | FDE(AI)+운영 | LLMOps/MLOps 실험·섀도배포 / AI엔지니어링·실험설계 |
| **D3 운영 전환** | SLA 안에서 안전 가동 | 안정화 2~4주 SLA위반·중대장애 0건 | 운영 Lead+FDE | 운영자동화·변경관리·롤백 / SRE·변경관리 |
| **D4 내재화** | 자동화 내재화 + 현장자산→재사용자산 | 목표 자동화율 달성, Asset N건 등록 | FDE+④플랫폼팀 | 자산카탈로그·오케스트레이션 / 자동화·자산화 |
| **D5 확장** | 같은 고객 내 새 유스케이스·과금 상향 | 추가 유스케이스 계약 또는 성과과금 상향 합의 | 영업+FDE Lead | 성과리포팅 / 고객성공·컨설팅 |
| D6 (상시) 운영·품질 | SLA·장애·변경·모델품질 상시 관리 | — | 운영+FDE | C-2~C-5 참조 |

### 4.3 게이트 — PoC 함정 탈출의 핵심
**G1. PoC → 운영 전환 게이트** (임계치는 예시·가정, 버티컬별 조정)

| 차원 | 통과 임계치(예시·가정) | 취지 |
|------|----------------------|------|
| ① 성과 | MTTR 20%↓ **또는** 티켓 자동분류율 30%↑ | 운영 가치 실재 확인 |
| ② 정확도·신뢰 | 정확도 ≥ 합의치, 오탐률 ≤ 임계치 | 오탐 많으면 운영 부하↑ |
| ③ 운영 안정성 | 섀도/카나리 중 중대장애·SLA 영향 **0건** | 안정성 훼손 금지 |
| ④ 경제성 | ROI 양(+) 또는 회수기간 합의 범위 | 운영에서 적자면 중단 |
| ⑤ 운영 수용성 | 운영팀 "운영 가능" 확인, 런북 정비 | 사람이 못 돌리면 실패 |
| **판정** | ①②③ 필수 충족 + ④⑤ 합의 → 통과 / 보류 / 중단 | — |

> 보류 시 1회 한 재PoC, 2회 연속 미달 시 자동 중단 룰. 버티컬 조정: 금융(안정성·오탐 더 엄격), 제조(가용성·MTTR 비중↑), 공공(변경통제·감사추적 필수).

**G2. 확장·성과과금 전환 게이트**: 운영 후 N개월(예 3개월) 연속 목표 KPI 유지 + 자동화 내재화 + 고객 만족 → 성과과금 전환 협상 개시.

### 4.4 영업-운영 연계 & 품질 프로세스 (D6 상시)
- **C-1 영업-운영 연계**: G0 핸드오프 체크리스트·합동 착수회의 의무화, 성과→세일즈 자산 환류, 영업이 게이트 기준 사전 학습(과약속 방지).
- **C-2 SLA**: AI 적용이 기존 SLA를 개선하되 절대 훼손 금지(악화 시 즉시 롤백). AI 자체에도 SLA 부여.
- **C-3 장애대응**: 섀도→사람승인→조건부자동 3단계 안전 상향. AI 오작동 시 에스컬레이션·롤백.
- **C-4 변경관리(규제 핵심)**: 모델/프롬프트 변경을 ITSM 변경통제에 등록·승인, 버전·감사로그, 드리프트 관리.
- **C-5 모델·데이터 품질**: 정확도·오탐·드리프트 상시 대시보드, GenAI 환각·민감정보 마스킹.

### 4.5 성과 정산식 (Outcome 과금)
```
성과보수 = Σ[(KPI_baseline − KPI_current) × 단위가치 × 귀속계수] × 성과배분율
          (단, floor ≤ 성과보수 ≤ cap)
```
- **분쟁방지(계약 사전 정의)**: D1 서명 베이스라인 고정, 월 측정·분기 정산, 귀속계수로 외부요인 분리, floor(공급사 최소수익)·cap(고객 과지급 방지).
- *식·수치는 설계 예시(가정). 고객·버티컬·실데이터로 확정.*

## 5. 툴·플랫폼

### 5.1 한 줄 원칙 — "고객 환경에 얹는다(adapt), 종속시키지 않는다(no lock-in)"
- 자사 단일 플랫폼을 강요하지 않고, 고객의 기존 멀티벤더 운영툴 위에 AI를 적용한다. 이것이 Palantir형 product-led와의 근본 차이이며, ①축 차별화의 기술적 구현이다.

### 5.2 기술 스택 4레이어 (상용+오픈소스 조합으로 고객 제약 대응)
| 레이어 | 역할 | 상용 예시 | 오픈소스 예시 | 비종속 구현 키 |
|--------|------|----------|--------------|---------------|
| L4 운영 자동화 | 티켓·런북·셀프힐링 | ServiceNow, PagerDuty | StackStorm, Rundeck, Ansible, n8n | 고객 ITSM·자동화 도구에 API/웹훅 연동 |
| L3 GenAI Ops/LLMOps | 장애대응 보조·런북생성·운영에이전트 | Azure OpenAI, Bedrock, Vertex | Llama·Qwen+vLLM, LangChain, Langfuse | **모델 게이트웨이**로 클라우드↔온프레미스 교체 |
| L2 AIOps | 이상탐지·RCA·예측 | Dynatrace, Datadog, BigPanda | Prometheus+이상탐지, PyOD, Keep | 알람·이벤트 표준 스키마로 엔진 교체 |
| L1 데이터·관측성 | 로그·메트릭·트레이스 수집 | Splunk, Datadog, Elastic | OpenTelemetry, Prometheus, Grafana | **OpenTelemetry 표준**으로 백엔드 교체 자유 |
| L0 보안 거버넌스 | 분류·접근통제·감사·마스킹 | (고객 IAM·DLP·SIEM) | OPA, Vault, Presidio | 고객 보안정책에 종속(우회 안 함) |

**비종속을 가능케 하는 2개 표준 (설계 결정)**
1. **수집은 OpenTelemetry(OTel)로 통일** — CNCF 사실상 표준. 백엔드(Splunk/Datadog/Grafana)를 고객 사정에 맞게 교체 가능 = "고객 환경 적응형"의 핵심 장치.
2. **모델은 게이트웨이로 추상화** — LLM 호출을 게이트웨이(LiteLLM 등) 뒤에 두어 클라우드 모델과 온프레미스 오픈모델을 코드 변경 없이 교체. 망분리·공공 환경 대응의 표준 패턴.

### 5.3 핵심 무기 — 재사용 Asset(액셀러레이터) ★
```
현장(FDE) 커스텀 자산 생성 → 자산 카탈로그 등록 → 코어팀(CoE) 심사·표준화
        ↑                                              ↓
  다음 고객에 재배포  ←  재사용(adapt)  ←  표준 자산 라이브러리(paved highway)
```
- 재사용이 늘수록 신규 고객 온보딩 공수가 줄어 1인 커버리지가 커진다 → **성과기반 과금의 마진을 떠받치는 구조**.
- 자산 유형: 데이터·관측성 템플릿 / AIOps 룰셋·플레이북 / LLMOps 패턴 / 자동화 런북 / 버티컬 레퍼런스 아키텍처 / KPI·정산 템플릿.
- *"템플릿 20종·재사용률·공수절감률"은 목표/추정치 — PoC 후 측정·갱신.*

### 5.4 규제산업 대응 아키텍처 — "망분리·온프레미스·데이터 주권"
- **데이터 주권**: 고객 운영데이터·로그·개인정보는 고객 경계(망) 밖으로 나가지 않는다. 망분리 환경은 온프레미스 오픈모델로 GenAI 구성(외부 API 호출 없음).
- **금융**: 전자금융감독규정 망분리 → 내부망 온프레미스 배포 가능 도구만 선정.
- **공공**: CSAP·국정원 요건 → SaaS 전용 배제, 국산·오픈소스·프라이빗 우선.
- **제조**: OT/폐쇄망 → 에지/온프레미스 경량·비침투 모니터링.
- *구체 제품 선정은 PoC 후 확정(비종속 원칙). 본 문서 제품은 예시(option).*

## 6. 역량·인력양성·CDP

### 6.1 복합역량 5대 영역 (Competency Model)
| 영역 | 정의 |
|------|------|
| A. 운영 도메인 | MSP 운영을 아는 힘(장애·티켓·SLA 현장 경험) |
| B. AI/데이터 | 운영에 AI를 얹는 힘(AIOps·GenAI/LLMOps·자동화) |
| C. 컨설팅/고객소통 | 문제 정의·성과 설득(Discovery·경영진 보고) |
| D. 엔지니어링 기본기 | 만들고 통합(코드·API·관측성·보안) |
| E. 자산화/방법론 | 한 번 해결한 것을 재사용 자산으로(차별화 마진의 원천) |

- 5영역 × 4레벨(Associate/FDE/Senior/Lead) 기대수준을 역량사전으로 정의 → 채용·교육·평가의 공통 잣대. **레벨 체계는 ② 조직 축과 1:1 정렬(검수 완료)**.

### 6.2 인력양성 — 내부 전환 중심
- **주력 트랙: "운영 엔지니어 → AX FDE"**. 운영 도메인 지식은 현장 경험으로만 쌓여 단기 습득이 어렵고, AI 툴은 표준 교육으로 단축 가능 → "운영 경험자에게 AI를 얹는" 것이 가장 빠르다(업계 통설 기반 추정, 1기 수료 후 실측 검증 권고).

| 출발점 | 채워야 할 갭 | 도착 | 양성 기간(추정) |
|--------|-------------|------|----------------|
| 운영 엔지니어(주력) | B.AI/데이터, C.컨설팅 | FDE | 약 6개월(교육3+OJT3) |
| SI 개발자 | A.운영, C.컨설팅 | FDE | 약 6~9개월 |
| 데이터/AI 인력 | A.운영, C.컨설팅 | FDE | 약 6~9개월 |
| 신규 시니어 채용 | 사내 방법론·고객 컨텍스트 | Senior FDE | 약 2~3개월 온보딩 |

**6개월 커리큘럼**: 1단계 기초(1개월) → 2단계 실습·모의 PoC(2개월, 캡스톤) → 3단계 현장 OJT(3개월, 멘토 사인오프 → FDE 인증).

> **[로드맵 제약 — 양성 처리량]** 1기 전환에 최소 6개월 소요 → Phase 1에는 외부 시니어 채용 + 1기 교육 병행만 가능. **현장 투입 가능 FDE 규모는 Phase 2(6~12개월)부터 본격화**. 사업 확산(Phase 3) 속도는 양성 처리량(throughput)에 의해 제약된다. (7장 로드맵에 반영)

### 6.3 CDP — 듀얼 래더 (리텐션 핵심)
```
                                  ┌─ [기술 트랙] FDE Architect (최상위 전문가)
Associate ─> FDE ─> Senior FDE ──┤
(0~1년)   (1~3년)  (3~6년)        └─ [관리 트랙] FDE Lead → 조직장
```
- **기술/관리 듀얼 트랙은 직급·보상 대등**(Google/MS의 IC 트랙 통설 차용) → 우수 엔지니어가 관리직으로 가지 않아도 성장·보상받게 해 이탈을 막는다.
- 인증·뱃지: AX FDE Certified → Senior → Architect/Lead Certified. 영역 뱃지(AIOps·GenAI·Observability·자동화·도메인). 외부인증(AWS/Azure/GCP Pro, CKA, ITIL 4)은 보조 증빙.

### 6.4 평가·보상 — 맨먼스 탈피, 성과 연동
| 평가 축 | 측정 지표 | 수익모델 연결 |
|---------|----------|--------------|
| ① 운영성과 | 담당 고객 MTTR 단축률·자동화율·운영비 절감 | 성과기반 과금 직접 근거 |
| ② 고객가치 | NPS/만족도, 업셀·갱신 기여 | Land-and-Expand 매출 |
| ③ 자산화 기여 | 재사용 Asset 등록·재사용 횟수 | 한계비용↓·마진 개선 |

- 고정급(레벨) + 변동급(성과). 팀 성과 + 개인 기여 혼합. **자산화 보너스**(타 현장 재사용 시 원작자 보상). **맨먼스 KPI 폐지** — 자동화로 1인 커버리지를 늘릴수록 보상받는 구조.
> 운영 KPI를 보상에 연동하려면 객관·자동 측정이 전제 — ③ 정산식·④ 측정 대시보드와 공동 확정(검수 완료).

## 8. 통합 KPI 트리

> 구조: **사업성과(최종) ← 운영성과(가치증명) ← 선행지표(원인)**. 아래로 갈수록 먼저 움직이는 선행지표다. 한 지표를 고객 보고·내부 평가·성과 정산에 공용으로 쓴다(②축 KPI 일원화).

```
[사업성과 — 경영진 최종 지표]
 ├─ AX FDE 매출 / 영업이익률
 ├─ 성과기반 계약 비중 (목표/추정)
 └─ 인당 매출 / 인원 대비 고객 수(커버리지 효율)
        ▲ (가치로 증명되어)
[운영성과 — 고객 가치 증명 / 공용 지표]
 ├─ MTTR 단축률            (③④⑤ 공통)
 ├─ 티켓 자동화율           (③④ 공통)
 ├─ 운영비 절감액           (① 정산 근거)
 ├─ SLA 준수율 / AI 기인 장애 0건 유지 (③ 품질)
 └─ 알람 노이즈 감축률 / 자동복구율 (④)
        ▲ (이것이 쌓여서)
[선행지표 — 가장 먼저 움직이는 원인 지표]
 ├─ PoC→운영 전환율(G1)    ★ "PoC 함정" 핵심지표 (③)
 ├─ G2 통과 후 업셀 건수     (③ 확장)
 ├─ 재사용 Asset 수 / 재사용률 / 온보딩 공수 절감률 (④)
 ├─ 양성 완료 FDE 수 / 전환 수료율 / 투입 리드타임 (⑤)
 ├─ 핵심인재 유지율 / 자발이직률 (⑤ 리텐션)
 └─ 등대고객 수 / 게이트 평균 소요기간 (①③)
```

**Phase별 KPI 초점 (선행 → 후행으로 성숙)**
| Phase | 주력 KPI |
|-------|---------|
| P1 | 선행: 양성 1기 착수, 자산 카탈로그 v1, 버티컬 확정, 등대고객 PoC 2건 |
| P2 | 운영성과 입증: G1 전환율, MTTR↓·자동화율↑, 양성 완료 FDE 수 |
| P3 | 사업성과 전환: 성과기반 계약 비중↑, 업셀 건수, 재사용률·커버리지 효율 |
| P4 | 마진·효율: 영업이익률, 인원 대비 고객 수, 핵심인재 유지율 |

> 정량 목표치(예: 성과계약 비중 X%, 전환율 Y%)는 회사 예산·등대고객 실데이터 입력 후 확정한다(현재 미입력).

## 부록 B. 핵심 의사결정 포인트 (경영진 결정 필요)

1. **조직 신설 승인 — CoE+상주팀 하이브리드(옵션 C) + 초기 12명**: 독립 사업부(A)나 매트릭스(B)가 아닌 옵션 C로 갈지, 그리고 P1 핵심 12명 채용·전환 착수를 승인할지. (이것이 차별화 가설의 조직적 토대)
2. **타깃 버티컬 2개 vs 3개 확정**: 금융·제조·공공 중 P1 진입 우선순위. 3개면 현장팀·인원이 늘어난다(부록 A7).
3. **소싱 비율·예산 확정**: 내부 전환 vs 외부 채용 비율(권고 65:35±5)과 인건비 예산. 사내 전환 가능 인력 풀 규모가 양성 속도와 확산 속도를 결정한다(R3·R9).
4. **성과기반 과금 전환 시점·하방보호 수준**: 언제 혼합과금(2단계)→성과/구독(3단계)으로 갈지, floor(공급사 최소수익) 수준을 어디로 둘지. 수익성 리스크와 직결(R2).
5. **등대고객 지정 및 KPI 정량 목표 합의**: P1에 PoC 착수할 기존 운영고객 1~2곳과, G1 게이트 임계치·성과계약 정량 목표(현재 모두 예시·가정)를 실데이터로 확정.
