# AX FDE 사업 추진안 — ITO / Application MSP 운영시장 타깃 버전
### 타깃: ITO(IT 운영위탁) / Application MSP(애플리케이션 관리형 서비스) 운영시장
#### 부제: "FDE 기반으로 ITO 및 애플리케이션 관리형 서비스(Application MSP) 업무 자체를 혁신한다"

> **본 안의 대상 시장은 ITO(IT 운영위탁) + 애플리케이션 운영·유지보수(Application Management, SM) + 관리형 서비스(MSP)의 "운영시장"이다. SI(시스템 구축) 시장이 아니다.** 본 추진안은 새 시스템을 "짓는" 사업이 아니라, 이미 고객이 맡긴 "운영" 업무 자체를 FDE 방식으로 혁신하는 사업이다.
>
> **용어 풀이**
> - **ITO**(IT Outsourcing, IT 운영위탁) = 고객의 IT 인프라·시스템 운영을 외부 사업자가 대신 맡아 운영하는 것.
> - **Application MSP / SM**(Application Management / System Maintenance, 애플리케이션 관리형 서비스·유지보수) = 고객이 쓰는 업무 애플리케이션의 장애 처리·변경 요청·릴리스·결함 수정 등 "운영 단계"를 맡는 것. SI가 "짓기"라면 SM은 "운영·고치기"다.
> - **MSP**(Managed Service Provider, 관리형 서비스 사업자) = 위 운영 업무를 묶어 제공하는 사업자.
> - **AX FDE**(AI Transformation Forward Deployed Engineer) = 고객의 ITO·Application MSP 운영 현장에 **내재화**되어, AI(AIOps·GenAI)를 그 운영 업무에 적용하고, 운영 성과(MTTR·티켓 자동화율·운영비)로 가치를 입증하는 조직.
> - **Squad(스쿼드)** = 하나의 운영 혁신 과제를 끝까지 책임지는 4~7명의 교차기능(cross-functional, 여러 직무가 한 팀에 모인) 팀. 딜리버리의 기본 단위.
> - **CoE**(Center of Excellence, 전문성 중앙조직) = Asset·품질·방법론·양성을 본사에서 관리해 Squad에 공급하는 조직.
> - **AIOps**(AI for IT Operations) = AI로 장애를 탐지·원인분석·자동복구하는 운영 기술.
> - **MTTR**(Mean Time To Repair/Resolve, 평균 장애복구시간) = 장애가 나서 정상화될 때까지 걸리는 평균 시간. 운영시장의 핵심 성과지표.
>
> 본 추진안은 5개 축(① 전략·차별화 / ② 조직·거버넌스 / ③ 프로세스·방법론 / ④ 툴·플랫폼 / ⑤ 역량·인력양성·CDP)을 하나의 논리(전략→조직→프로세스→툴→사람→로드맵)로 통합한 것이다.

## 1. Executive Summary

> **선언 (본 안의 범위):** 본 추진안의 대상은 **ITO/Application MSP 운영시장**(IT 운영위탁 + 애플리케이션 운영·유지보수 + 관리형 서비스)이며, **SI(시스템 구축) 시장이 아니다.** 우리는 새 시스템을 짓는 사업을 하려는 것이 아니라, 고객이 이미 우리에게 맡긴 **"운영" 업무 자체를 FDE 방식으로 혁신**하려는 것이다.

### 왜 (운영시장의 Pain Point — SI가 아니라 운영의 문제다)
운영시장의 현재 모델은 **"상주 인력 단가(맨먼스) 기반 과금"**에 묶여 있다. 즉, 고객 현장에 사람을 몇 명 상주시키느냐로 매출과 비용이 정해진다. 여기서 다음 Pain Point가 동시에 발생한다.

- **상주 맨먼스 과금의 구조적 한계**: 매출이 상주 인원 수에 비례한다. 운영을 더 잘해서가 아니라 사람을 더 넣어야 매출이 는다. 마진과 확장성이 막힌다.
  - *맨먼스(Man-Month) = 인력 1명이 1개월 일하는 작업량 단위. 운영시장의 전통 과금 기준.*
- **운영비 상승 + 인건비 상승·인력난**: SW기술자 일평균임금은 2025년 41.5만원으로 전년 대비 +4.7%(2023년 +8.6%)다(출처: SW산업협회/IITP). 운영인력을 채우기도 어렵고, 채워도 단가가 매년 오른다. 상주 인원에 매출이 묶인 구조에서는 인건비 상승이 곧 마진 잠식이다.
- **SLA 압박과 장애대응의 인력 의존**: 고객은 SLA(Service Level Agreement, 운영 품질 약속)를 점점 빡빡하게 요구하는데, 장애 탐지·원인분석·복구가 여전히 "사람의 눈과 손"에 의존한다. 야간·주말 장애 대응을 위해 더 많은 상주·당직 인력이 필요해진다.
- **반복 운영업무의 비효율**: 티켓 분류, 단순 장애 1차 대응, 반복 요청 처리, 로그 뒤지기 등 부가가치 낮은 반복 업무에 숙련 인력의 시간이 소모된다.

> **핵심 진단:** 운영시장의 Pain Point는 "구축을 잘 못해서"가 아니라 **"운영을 사람 머릿수로 한다"**는 데서 온다. 따라서 해법도 운영시장의 것이어야 한다 — 운영 업무 자체를 AI로 혁신하는 것이다.

### 무엇을 (AX FDE — 운영 업무 자체의 혁신)
- 고객의 **기존 ITO/Application MSP 운영 현장에 Squad가 내재화**되어, AIOps·GenAI로 운영 성과를 낸다. 외부에서 컨설팅 보고서를 주는 것이 아니라, 운영 안으로 들어가 운영을 바꾼다.
- 혁신 대상은 운영 업무 4영역이다(3장 상세): **① 인프라 운영**(장애탐지·RCA·셀프힐링), **② 애플리케이션 운영/SM**(장애·요청 처리, 코드 진단, 릴리스), **③ 서비스데스크/티켓**(자동분류·자동응대), **④ 운영 자동화**(런북·플레이북).
- 딜리버리 기본 단위는 **Squad(4~7명 교차기능 팀)**다. 각 Squad가 **6~8주 타임박스**로 1개 운영 혁신 과제를 회전시켜 빠른 Proof of Value(PoV·가치입증)(time-to-value)을 만든다.
- 운영 성과로 가치를 입증한다: **MTTR 단축, 티켓 자동화율, 운영비 절감, SLA 개선, 알람 노이즈 감축.** 검증된 기술 위에 선다 — AIOps 도입 기업의 74%가 MTTR을 단축했고, GenAI 적용으로 운영 티켓 분류 시간이 44% 줄었다(출처: GlobalGrowthInsights, Workativ/Moogsoft 인용).
- **과금을 단계적으로 전환한다**: 상주 인력 단가(맨먼스) → 성과/구독 기반 과금. 운영을 잘할수록(사람을 덜 쓰고 성과를 낼수록) 더 버는 구조로 바꾼다.

### 어떻게 (차별화 — 전 축을 관통하는 가설)
> **"FDE형 운영혁신" — 고객의 기존 ITO/Application MSP 운영에 AI를 적용하되, 특정 플랫폼에 묶지 않고, 6~8주에 운영 성과를 보이고 그 성과를 운영에 내재화해, 성과로 값을 받는 모델.**

1차 비교 대상은 경쟁사가 아니라 **"전통 운영위탁(상주 맨먼스)" 모델 그 자체**다.

| 비교 축 | 전통 운영위탁 (상주 맨먼스) | 우리 (FDE형 운영혁신) | 어느 축이 구현하나 |
|----------|------------------------|-------------------|------------------|
| 매출 동인 | 상주 인원 수 | 운영 **성과**(MTTR↓·자동화율↑·운영비↓) | ① 전략 |
| 과금 | 상주 인력 단가(맨먼스) | **맨먼스 → 성과/구독 기반 단계 전환** | ① 전략 + ⑤ 평가보상 |
| 장애대응 | 사람의 눈·손에 의존(당직) | **AIOps 탐지·RCA·셀프힐링으로 인력 의존↓** | ④ 툴 + ③ 프로세스 |
| 반복 운영업무 | 숙련 인력이 직접 처리 | **티켓 자동분류·자동응대·런북 자동화** | ④ 툴 |
| SLA | 인력 증원으로 방어 | **AI로 MTTR·노이즈 개선해 방어** | ③ 프로세스(게이트·정산식) |
| 인건비 상승 | 그대로 마진 잠식 | **자동화로 흡수, Squad 커버리지 확대** | ④ 재사용 Asset + ② CoE |
| 확장 | 사람 더 뽑아야 확장 | **재사용 Asset로 다고객 확산(Land-and-Expand)** | ① 전략 + ② 조직 |

- **Palantir FDE는 "차용한 딜리버리 방법론"일 뿐, 경쟁 대상이 아니다.** 우리가 Palantir와 같은 시장에서 겨루는 것이 아니다. 우리는 Palantir의 **Squad 편제·6~8주 타임박스·Asset 재사용 루프**라는 **일하는 방식(딜리버리 방법론)만 차용·변형**해 ITO/Application MSP 운영혁신에 적용한다(상세 2.4).

### 핵심 — FDE를 ITO 운영에 적용한다는 것의 의미
고객의 운영 현장에 Squad가 들어가, 6~8주 타임박스로 운영 과제를 빠르게 입증하고(예: "장애 자동탐지로 MTTR 25% 단축"), 통과하면 그 결과를 운영에 내재화한 뒤, 같은 Asset을 다음 고객으로 확산한다. 즉 **빠른 입증 → 운영 내재화 → 다고객 확산(Land-and-Expand)**이다. (상세 3장)

### 기대효과 (KPI — 운영시장 지표)
- **사업성과**: AX FDE 매출, 영업이익률, **성과기반 계약 비중**, Squad당 분기 과제 처리량.
- **운영성과(가치증명)**: MTTR 단축률, 티켓 자동화율, 운영비 절감액, SLA 준수율, **AI 기인 장애 0건**, 알람 노이즈 감축률, 자동복구율.
- **선행지표**: G1 과제→운영 전환율, 가동 Squad 수, 양성 완료 수, 재사용 Asset 수.

### 권고 (경영진 결정 요청)
- 중견~대형 MSP/ITO 기준 **약 50명·6 Squad**(확정), **버티컬 2~3개**(금융·제조·공공), **기존 운영고객 1~2곳 Lighthouse Customer(등대고객 — 성과를 먼저 입증해 다른 고객으로의 확산 기준이 되는 첫 도입 고객)**에서 6~8주 과제 → 운영 내재화 → 성과과금 전환 경로를 탄다.
- 4-Phase 로드맵: P1 기반(0~6M, 약 19명·2 Squad) → P2 파일럿(6~12M, 약 35명·4 Squad) → P3 확산(12~24M, 약 50명·6 Squad) → P4 고도화(24M~).

> *주: 인원 50명·Squad(4~7명)·6~8주 타임박스·로드맵 2→4→6 Squad는 **"확정 입력"**. 버티컬 2~3개·예산·현 인력 구성·게이트 임계치는 **"가정/추정"**으로 진행하며 실데이터 입력 시 갱신한다.*

## 2. 시장·전략·차별화

### 2.1 시장 정의 — "성장하는 운영시장, 그러나 맨먼스 모델 전환기"
- **규모/성장 (운영·MSP 시장)**: 국내 클라우드 MSP 운영시장은 2023년 약 7조원 → 2026년 약 12조원, 연 15% 전망(가트너). 일부 전망은 5년 연평균 30%. *출처 간 편차가 커 "약/추정" 표기.* (출처: 디지털데일리·머니투데이 2025.01, IDC 인용)
- **경쟁구도**: IDC 평가 국내 MSP 1위 삼성SDS 23.9% — 대형 SI·보안기업의 운영시장 진입으로 "대경쟁 시대".
- **구조변화(운영시장의 전환 압력)**: 2025년 SW기술자 일평균임금 41.5만원, 전년 +4.7%(2023년 +8.6%). **상주 맨먼스 단가 상승 + 운영 인력난** → 상주 인원에 매출이 묶인 운영위탁 모델의 마진·확장성 한계.

> **본 안의 시장 한정:** 위 수치는 모두 **운영·관리형 서비스(MSP/ITO) 시장**의 것이다. SI(신규 구축) 발주 시장은 본 안의 대상이 아니다.

### 2.2 FDE 모델이 운영시장에 맞는 이유
- 한국 운영시장은 상주 맨먼스 중심 → 매출이 상주 인원 수에 비례 → 임금 상승 시 마진 압박. FDE는 매출을 **인원 수가 아닌 운영 성과**에 연동하고, 자동화로 Squad 커버리지를 키워 임금 상승을 마진으로 흡수한다.
- 기술 근거가 검증돼 있다: AIOps 도입 기업 74% MTTR 단축, GenAI 적용 운영 티켓 분류 44% 단축, 운영효율 63% 개선(출처: GlobalGrowthInsights, Workativ/Moogsoft). 이는 모두 **운영 단계**의 성과지표다.

### 2.3 차별화 — "FDE형 운영혁신" + 속도(time-to-value)
- 우리의 차별화는 **전통 운영위탁(상주 맨먼스) 대비**다: ① AIOps·GenAI로 장애·티켓·반복업무를 자동화해 **인력 의존을 줄이고** ② **운영 KPI**(MTTR·자동화율·운영비·SLA)로 입증하며 ③ 기존 운영계약을 고도화하는 **Land-and-Expand**로 시작하고 ④ **맨먼스 → 성과/구독 과금**으로 전환한다.
- **속도 차별화**: "수주 내 빠른 PoV"이라는 발상을 **6~8주 타임박스 + Squad**로 운영시장에 적용하되, 일회성 시연이 아니라 **G1 게이트로 운영 내재화에 연결**한다. "빠른 입증 → 운영 내재화 → 성과과금 확대"가 속도-지속성 결합 메시지다.

### 2.4 Palantir FDE에 대한 위치 정의 — "방법론의 출처, 경쟁 대상 아님"
> **중요:** Palantir FDE는 우리의 경쟁 시장이 아니다. Palantir는 **자사 플랫폼(Foundry) 확산을 위한 product-led 모델**로, 우리가 다루는 ITO/Application MSP 운영시장과 다른 영역이다.

우리는 Palantir에서 **딜리버리 방법론(일하는 방식)만 차용·변형**한다:

| 차용 요소 | Palantir 원형 | 우리의 변형(운영시장 적용) |
|----------|--------------|--------------------------|
| Squad 편제 | 고가치 신규 프로젝트에 교차기능팀 투입 | 고객 **운영 현장**에 내재화되는 교차기능 Squad |
| 타임박스 | 수주 내 빠른 PoV | **6~8주 타임박스로 운영 KPI 입증** |
| Asset 재사용 루프 | 현장 커스텀("gravel road") → 코어팀 표준화("paved highway") → Foundry 확산 | 현장 운영 Asset → CoE 표준화 → **다고객 운영 재사용**(플랫폼 비종속) |
| 출발점 | 신규 고가치 프로젝트 | **기존 운영계약 고도화(Land-and-Expand)** |

- 핵심 차이: Palantir는 방법론으로 **자사 제품을 판다**. 우리는 같은 방법론으로 **고객의 운영을 혁신**한다. 따라서 차별화 표(2.3·1장)의 1차 비교는 "전통 운영위탁 vs 우리"이고, Palantir는 위처럼 **방법론 차용·변형 대비**로만 다룬다.

### 2.5 경쟁 포지셔닝
> "우리는 사람을 상주시켜 운영비를 받는 전통 운영위탁이 아니다. 고객 운영 현장에 들어가 **AI로 운영 성과를 내고 성과로 값을 받는** 운영혁신 파트너다."

### 2.6 수익모델 — 상주 맨먼스 → 성과기반 3단계 (사이클 정합)
| 단계 | 과금구조 | 성과정산 | 전제 게이트 |
|------|---------|---------|------------|
| 1단계 | 상주 맨먼스(M/M) | 없음(현행 운영계약) | 착수(G0) |
| 2단계 | 맨먼스 + 성과 인센티브 | 운영 개선분 일부를 인센티브로(소액·검증) | **6~8주차 G1 통과** |
| 3단계 | 성과/구독 | 정산식 본격 적용 | 상시 트랙 **G2(지속성과)** |

- 6~8주 G1 통과 시점에 2단계로 전환해 빠른 PoV와 과금을 연결한다. 짧은 사이클로 과금 전환의 검증 빈도를 높인다.

### 2.7 GTM — "기존 운영계약 고도화 → 6~8주 입증 → 운영 전환 → 다수 Squad 확산"
- 신규 영업이 아니라 **기존 ITO/Application MSP 운영계약의 AI 고도화(Land-and-Expand)**로 시작. 이미 운영을 맡고 있어 데이터·접근권·신뢰가 확보돼 있다는 점이 운영시장 특유의 강점이다. Lighthouse Customer 1~2곳에 6~8주 타임박스 과제 → 운영 성과 베이스라인 입증 → 운영 전환 → 다수 Squad로 버티컬 동시 공략.

### 2.8 수익모델 심화 — 플랫폼·모델 비보유 기업의 FDE 수익 엔진 (Palantir·OpenAI 비교)

> **왜 이 절이 필요한가**: Palantir·OpenAI의 FDE는 "현장 엔지니어 인건비를 **원가에 녹이고**, 수익은 자기가 소유한 **복제 불가 자산**(플랫폼·모델)의 **반복 소비**로 회수"하는 구조다. 우리처럼 **선진 LLM(frontier model·최상위 성능의 자체 거대언어모델)도, Palantir급 독점 솔루션도 없는 회사**는 같은 구조를 그대로 복제할 수 없다. 그렇다면 무엇을 수익 엔진으로 삼아야 하는가를 정리한다.

**(1) 선도사 수익모델 해부 — FDE는 비용, 진짜 수익은 "소유 자산의 반복 소비"**

| 구분 | Palantir | OpenAI·Anthropic |
|------|----------|------------------|
| FDE의 위치 | **비용**(원가·판관비)으로 흡수 — AIP 계약을 맺은 고객에 투입돼 파일럿 만료 전 ontology(업무·데이터 개념모델)를 현실 데이터에 연결 | **비용**으로 흡수 — 엔터프라이즈에 내재화해 커스텀 구축, 토큰 소비를 끌어올림 |
| 진짜 수익 엔진(moat·해자) | 자사 플랫폼 **Foundry·AIP 라이선스·사용량**(복제 불가) | **frontier 모델 + API 토큰 소비 + 엔터프라이즈 시트·SLA** |
| 비유 | **razor-and-blade(면도기-면도날)**: FDE가 면도기를 깔고, 돈은 면도날(플랫폼 소비)에서 | 동일 — FDE가 깔고, 돈은 토큰 소비에서 |
| 결과 | "컨설팅처럼 보이되 software처럼 확장", 美 상업매출 +137% YoY(’25 4Q, 출처 보고) | 토큰 단가는 약 98% 하락했지만 agentic(자율 에이전트) 소비 급증으로 엔터프라이즈 청구액은 오히려 약 3배 |

> **핵심 통찰(업계 정설)**: "**value metric(과금 기준)과 cost(원가)를 분리하라** — 고객은 성과에 값을 내고, 공급사는 원가를 내부에서 관리한다. 과금을 비즈니스 성과에 묶은 회사는 원가가 떨어질 때 그 차익을 **마진으로** 가져간다."

**(2) 진단 — 이 모델의 숨은 전제**

선도사 모델의 전제는 **"FDE 비용을 녹여 회수할, 내가 소유한 고마진·반복 수익 엔진(독점 자산)이 있다"**는 것이다. 그 자산이 없는 회사가 FDE만 하면 결국 **맨먼스 서비스로 회귀**한다(선형 매출·저마진). 따라서 비보유 기업의 과제는 단 하나 — **"우리만의 면도날(반복 수익 엔진)을 무엇으로 삼을 것인가"**이다.

**(3) 플랫폼·모델 비보유 기업의 수익모델 옵션 비교**

| 옵션 | 내용 | 수익 엔진(moat) | 마진·확장성 | 평가 |
|------|------|----------------|------------|------|
| A. 순수 FDE 서비스 | 인력 투입(맨먼스) | 없음 | 선형·저마진 | ✕ 회피 대상 |
| B. 선도사 플랫폼/모델 리셀러+통합 | 남의 면도기에 면도날 깔아주기 | **남의** 자산(종속) | 박마진·종속 리스크 | △ 보조만 |
| C. 성과기반 매니지드 서비스 | outcome(성과) 과금, 원가는 내부 관리 | 운영/구축 성과 데이터·정산식 | 원가하락분=마진 | ○ 핵심 축 |
| D. **자사 소유 중립 컨트롤 플레인 + 자사 CSP** | 보안·거버넌스·멀티LLM 게이트웨이 + 자사 클라우드 소비 | **우리 플랫폼·CSP의 반복 소비**(우리만의 면도날) | software형 확장·고마진 | ◎ 권고 |
| E. 버티컬 IP·Asset 제품화 | FDE 산출 Asset을 반복 판매(productized service) | 재사용 Asset·레퍼런스 | 한계비용↓ | ○ 보강 축 |

**(4) 권고 — 우리 자산으로 "면도날"을 재구성한다 (D 중심 + C·E 결합)**

우리는 frontier 모델로는 못 이긴다. 대신 **Palantir·OpenAI가 구조적으로 가질 수 없는 포지션**을 자산화한다.

- **그들의 약점 = 우리의 자리**: 단일 플랫폼/단일 모델 **lock-in(특정 공급사에 묶여 못 벗어나는 종속)**, 데이터 주권·규제(망분리) 한계. 엔터프라이즈는 lock-in을 두려워하고, 여러 모델에 걸친 **거버넌스·비용추적·감사**를 절실히 필요로 한다(토큰 청구액 급증이 이 수요를 키운다).
- **우리 보유 자산 = 대체 수익 엔진**: ① **자사 CSP**(데이터 주권·온프레미스·망분리 대응 가능한 자체 클라우드) ② **보안·거버넌스·멀티LLM 컨트롤 플레인**(control plane — 여러 LLM 호출을 하나의 정책·인증·예산·감사로 묶는 **중립 게이트웨이**). **멀티LLM 중립성 자체가 상품**이다 — 우리가 frontier 모델을 안 가졌기 때문에 **오히려 중립적일 수 있다**(선도사는 자사 모델로 유도할 유인이 있어 중립이 불가능).
- **수익 구조(razor-and-blade 재구성)**: FDE(설치·원가 흡수) → **자사 CSP + 거버넌스 플랫폼(우리 면도기)** → **멀티LLM 소비·거버넌스 구독·CSP 사용량(면도날, 반복 수익)**.
- **3대 수익 회수 경로**: ① 성과/가치 기반 과금(C) ② **자사 CSP 인프라 소비**(compute·storage) ③ **거버넌스·게이트웨이 플랫폼 구독+사용량**(시트·콜·토큰 라우팅). FDE 비용은 원가로 녹이고 이 셋으로 회수한다.
- **전략 정합**: 이는 ④ 툴축의 "플랫폼 비종속·OpenTelemetry·모델 게이트웨이" 원칙을 **수익모델로 승격**한 것이다 — 차별화(비종속)와 수익 엔진(우리 플랫폼·CSP 소비)이 **같은 자산**에서 나온다.

- **운영시장 적용**: 운영 워크로드 자체가 **자사 CSP + 거버넌스 플랫폼 위에서** 돌아가므로 반복 소비가 본질적으로 발생한다. FDE 운영혁신비는 원가로 녹이고, 수익은 ① 성과/구독 과금 ② **자사 CSP 소비**(운영 워크로드) ③ **멀티LLM 게이트웨이 사용량·거버넌스 구독**으로 회수한다. 운영의 **상시성**과 결합해 Palantir의 "라이선스 꼬리(반복 수익)"를 **우리식(중립·주권)**으로 재현한다.

**(5) 샘플 매출 시뮬레이션 (참조용 — 모든 수치는 예시·가정, 실데이터로 대체)**

> ⚠️ 아래는 **모델 구조를 보여주기 위한 가상의 예시**다. 단가·회전·마진은 임의 가정값이며 **의사결정 근거로 쓰지 말 것**. 실제 CSP 단가·플랫폼 과금·과제 단가 입력 시 대체한다.

*가정 (예시)*

| 가정 항목 | 예시값 | 비고 |
|----------|-------|------|

| 고객당 연 거버넌스·멀티LLM 게이트웨이 | 1.0억원 | 구독(시트) + 사용량(콜·토큰 라우팅) |

| 마진 (예시) | 서비스 30% / 반복(CSP+플랫폼) 65% | 반복 수익이 고마진 |

*Phase별 매출 구성 (억원, 예시)*

| 구분 | P1 (2 Squad·누적 2고객) | P2 (4 Squad·누적 6) | P3 (6 Squad·누적 12) |
|------|:--:|:--:|:--:|
| 운영 서비스 (FDE, 면도기) | 20 | 40 | 60 |
| 자사 CSP 소비 (반복, 면도날) | 3 | 9 | 18 |
| 거버넌스·게이트웨이 (반복, 면도날) | 2 | 6 | 12 |
| **합계** | **25** | **55** | **90** |
| 반복 수익 비중 | 20% | 27% | **33%** |
| 블렌디드 마진 (예시) | 37% | 40% | **42%** |
| (대조) 순수 맨먼스 시 | 매출 ~20·마진 ~30% | ~40·~30% | ~60·~30% |

**읽는 법**: 운영 서비스(면도기)는 가동 Squad 수에 비례하지만, **자사 CSP·거버넌스 플랫폼 소비(면도날)는 고객이 누적되며 쌓여** 반복 수익 비중이 20%→33%로 오르고, 고마진 반복이 섞이며 블렌디드 마진도 37%→42%로 개선된다. **순수 맨먼스(반복 없음·마진 정체)와의 차이가 곧 이 모델의 값**이다.

> **근거·한계**: Palantir FDE·AIP/Foundry 수익구조와 美 상업매출 성장, OpenAI·Anthropic의 FDE 채택·엔터프라이즈 과금, 토큰 단가 하락 대비 엔터프라이즈 청구액 증가, 멀티LLM 게이트웨이(컨트롤 플레인)의 거버넌스 수요는 공개 자료에 근거한다(출처: MindStudio, Everest Group, The Next Web, MuleSoft·TrueFoundry AI Gateway 자료, Bessemer·McKinsey AI 수익화 분석). 인용 수치는 시점·출처별 편차가 있어 "약/추정" 표기. **우리 CSP·플랫폼 기준 매출·마진·소비단가는 미입력 — 실데이터로 확정(추정)**.

| 수익모델 | 맨먼스→성과 과금 3단계·성과지표 계약화 | 전략+재무 | P1~P2 |
| 수익 엔진 | 자사 CSP+거버넌스·멀티LLM 플랫폼을 FDE 수익 엔진으로 설계(수익모델 심화 절) | 전략+플랫폼 | P1~P2 |

## 3. FDE의 ITO/Application MSP 운영 적용 (핵심)

> 이 장이 본 안의 심장이다. "FDE를 ITO 운영에 적용한다"가 구체적으로 무슨 업무를 어떻게 바꾸는지를 정의한다.

### 3.1 적용 원리 — 운영 현장 내재화
- Squad는 고객의 **기존 운영위탁/애플리케이션 운영 현장에 내재화**된다. 외부 컨설턴트로 방문하는 것이 아니라, 운영팀의 일부로 들어가 그들의 티켓·장애·로그·릴리스 흐름 안에서 AI를 적용한다.
- 적용 결과는 운영팀이 계속 쓸 수 있어야 한다(운영 수용성). 그래서 모든 과제는 런북·플레이북·대시보드로 운영에 남는다.

### 3.2 ITO 운영 혁신 4영역 (무엇을 바꾸는가)

#### 영역 1 — 인프라 운영 (Infrastructure Operations)
- **현행 Pain Point**: 알람 폭주(노이즈), 사람이 직접 장애 탐지·원인분석, 야간·주말 당직 의존.
- **FDE 적용**: AIOps 기반 **이상 탐지 → RCA(Root Cause Analysis, 장애 근본원인 자동분석) → 셀프힐링(self-healing, 사전 정의된 조치로 자동 복구)**. 알람 상관분석으로 노이즈를 묶고, 반복 장애는 자동 복구로 사람 호출 자체를 줄인다.
- **성과지표**: MTTR 단축, 알람 노이즈 감축률, 자동복구율, 야간 호출 건수↓.

#### 영역 2 — 애플리케이션 운영 / SM (Application Management) ★ 본 안의 보강 영역
- **현행 Pain Point**: 애플리케이션 장애·결함·변경요청(SR)이 사람 손에 묶여 처리량 한계, 레거시 운영지식이 특정 인력에 key-person dependency(속인성 — 업무가 특정 사람에게만 묶이는 것), 코드·로그 분석에 숙련 시간 소모.
- **FDE 적용 (GenAI 중심)**:
  - **GenAI 코드 진단·로그 분석**: 장애 발생 코드 구간·예외 로그를 LLM이 1차 분석해 후보 원인·관련 코드·유사 과거 이슈를 제시 → 운영 개발자의 진단 시간 단축.
  - **운영 중 결함 대응 자동화**: 반복 결함 패턴을 탐지해 임시조치(완화책)·재현 절차·패치 후보를 제안.
  - **변경/릴리스 관리 보조**: 변경 영향도 분석, 릴리스 노트·테스트 케이스 초안 생성, 배포 전 체크리스트 자동 점검.
  - **레거시 운영 지식의 RAG화**: 운영 매뉴얼·과거 장애이력·코드 주석·위키를 **RAG**(Retrieval-Augmented Generation, 사내 문서를 검색해 답하는 AI)로 묶어, key-person dependency 상태의 운영지식을 팀 Asset으로 전환.
- **성과지표**: SR/결함 처리 리드타임 단축, 코드 진단 시간 단축, 운영지식 검색 응답 정확도, 릴리스 사고율↓.

#### 영역 3 — 서비스데스크 / 티켓 (Service Desk & Ticketing)
- **현행 Pain Point**: 티켓 수기 분류·라우팅, 반복 문의 1차 응대에 인력 소모.
- **FDE 적용**: GenAI 기반 **티켓 자동분류·우선순위 산정·자동 라우팅**, 반복 문의 **자동응대(셀프서비스)**, 유사 과거 티켓 추천.
- **성과지표**: 티켓 자동화율(GenAI 적용 시 분류시간 44% 단축 — 출처 부록 C), 1차 응대 자동처리 비율, 평균 응답시간 단축.

#### 영역 4 — 운영 자동화 (Runbook & Playbook Automation)
- **현행 Pain Point**: 반복 운영작업이 수작업 또는 단순 스크립트에 머묾, 절차가 문서에만 존재.
- **FDE 적용**: 반복 운영절차를 **런북/플레이북으로 코드화**하고 ITSM·자동화 도구에 연동해 조건부 자동실행. 사람 승인 단계를 거쳐 안전하게 자동화 수준을 상향.
- **성과지표**: 자동화된 운영작업 수, 수작업 공수 절감률, 자동복구율.

### 3.3 PoV → 내재화 → 확산 (Land-and-Expand)
```
[6~8주 타임박스 — 빠른 운영 과제 입증]
   인프라/앱/티켓/자동화 영역에서 1개 과제 선택
   → 운영 KPI 베이스라인 측정 → AI 적용 → G1 판정
        │ 통과
[운영 내재화 — 상시 트랙]
   런북·플레이북·대시보드로 운영에 남김 → G2(지속 성과)
        │
[다고객 확산]
   CoE가 표준화한 Asset을 다음 운영고객에 재배포(adapt)
```
- **6~8주에 운영 과제를 빠르게 입증**한다(예: "장애 자동탐지 도입으로 MTTR 25% 단축", "티켓 자동분류율 35% 달성").
- 통과 과제는 **운영에 내재화**(운영팀이 계속 쓰는 Asset으로 정착)된다.
- 같은 Asset을 **다고객으로 확산**해 Marginal Cost(한계비용 — 1건을 더 할 때 드는 추가 비용)을 낮춘다.

### 3.4 과금 전환 연계
- 운영 성과가 입증될수록 상주 맨먼스 비중을 줄이고 성과/구독 비중을 늘린다(2.6 3단계). 운영을 잘할수록(사람을 덜 쓰고 성과를 낼수록) 더 버는 구조다.

## 4. 조직·거버넌스

### 4.1 조직 형태 — "Squad + CoE + 리더십" (약 50명, 확정 입력)
- 6~8주 타임박스로 운영 과제를 회전시키려면 부서 간 협업을 매번 조율할 시간이 없다. **한 팀 안에 진단(컨설턴트)·운영도메인(업종전문가)·설계(아키텍트)·구현(AI/Data Eng)이 다 있어야** 빠르게 끝낸다 → 교차기능 Squad가 답이다.
- Squad 산출물(런북·룰셋·RAG·대시보드)을 **CoE가 운영 Asset으로 흡수**(표준화·재사용)하면 다음 Squad는 처음부터 만들지 않아 회전이 빨라진다.

| 구성 | 역할 | 인원(50명) | 위치 |
|------|------|:---:|------|
| **Squad (딜리버리)** | 6~8주 타임박스 운영 과제 수행, 고객 단위 배정 | 약 34명 (6 Squad) | 고객 운영 현장(상주/하이브리드) |
| **CoE (중앙)** | 운영 Asset·품질·방법론·양성, Squad에 Asset 공급 | 약 11명 | 본사 |
| **리더십·사업개발** | 손익·전략, 업셀·영업운영 연계 | 약 5명 | 본사 |

### 4.2 50명 산정 근거 (확정 입력의 구성)
| 구분 | 인원 | 산정 근거 |
|------|:---:|------|
| 리더십·사업개발 | 5 | 사업총괄 1 + 사업개발/Pre-Sales 4 |
| CoE | 11 | 아키텍처 챕터 2 + AIOps/Platform 3 + 품질·방법론 2 + 양성 2 + 데이터/Asset 2 |
| Squad 딜리버리 | 34 | 50 − 5 − 11 |
| **Squad 수** | **6개** | 34 ÷ 평균 5.7명 ≈ 6 Squad (Squad당 4~7명 가변) |

- Squad 골격 = 필수 역할 4(컨설턴트·업종전문가·아키텍트·AI/Data Eng) + Squad Lead = 5명. 애플리케이션 운영(SM)·데이터 무거운 과제는 +1~2명(최대 7), 진단 위주 초기 과제는 4명.
- 6 Squad × 분기 2~3회전 = 분기 약 12~18개 운영 과제 처리(추정). 이 처리량이 다고객 확장의 엔진.

### 4.3 조직도 (50명)
```
               AX FDE 사업 총괄 (헤드) 1명  — 손익·전략·임원보고
   ┌──────────────────────┼──────────────────────┐
 Squad 딜리버리 약 34명     CoE (본사) 약 11명       사업개발/Engagement 4명
 6 Squad (각 5~6명)        - 아키텍처 챕터 리드 2    - 사업개발 리드 1
 Squad 1 … Squad 6        - AIOps/Platform Eng 3   - Pre-Sales/PoC 3
 (+공통 온콜 풀)           - 품질·방법론 2
                          - 양성·인에이블 2
                          - 데이터/자산관리 2
   * 헤드1 + Squad34 + CoE11 + 사업개발4 = 50명
```
- **Squad = 고객 단위 배정**: 과제 기간(6~8주) 동안 고객 운영 현장에 상주/하이브리드, 종료 후 다음 고객·과제로 재배정.
- **챕터(Chapter)**: 같은 역할 패밀리(예: 아키텍트)는 Squad에 흩어져 있어도 CoE 챕터 리드가 기술표준·역량을 횡단 관리(Spotify형 Squad-Chapter 차용, CoE가 Chapter 관리 주체). 실선=Squad Lead, 점선=CoE 챕터 리드.

### 4.4 역할 패밀리 × 시니어리티 매트릭스 (⑤ 역량 축과 동일 체계, 확정 입력)
- **가로축(무슨 일)=역할 패밀리, 세로축(얼마나 숙련)=시니어리티**의 매트릭스로 직무를 정의한다.

**시니어리티 축 (4레벨)**

| 레벨 | 시니어리티 | 의미 | 경력(추정) |
|------|-----------|------|-----------|
| L4 | **Lead** | Squad Lead 자격, 딜리버리 총괄·운영전환 책임 | 10년+ |
| L3 | **Senior** | 역할 주도, 운영 현장 의사결정, 코칭 | 7~10년 |
| L2 | **FDE(미들)** | 역할 1인분 독립 수행 | 4~7년 |
| L1 | **Associate(주니어)** | 학습 트랙·보조(운영 전환 인력 첫 단계) | 0~4년/전환자 |

**역할 패밀리 (4종) × 시니어리티 = 16셀 직무**

| 시니어리티 ↓ \ 역할 → | 컨설턴트<br>(운영 진단·문제정의) | 업종 전문가<br>(금융·제조·공공 운영) | 아키텍트<br>(설계·비종속통합·자산화) | AI/Data Engineer<br>(AIOps·GenAI·파이프라인) |
|---|:---:|:---:|:---:|:---:|
| **L4 Lead** | ● Squad Lead 가능 | ● | ● | ● |
| **L3 Senior** | ● | ● | ● | ● |
| **L2 FDE** | ● | ● | ● | ● |
| **L1 Associate** | ● | ○ | ○ | ● |

- **Squad Lead는 직무가 아니라 역할**: 4개 패밀리 중 L3~L4 시니어가 과제 성격(인프라 운영 위주냐 애플리케이션 운영 위주냐)에 따라 맡는다.
- **⑤ 역량 축 정렬**: 역량모델·평가·CDP는 이 4 패밀리 × 4 시니어리티 = 16셀을 직무 단위로 삼는다.

### 4.6 채용·소싱 — 역할 패밀리별 차등 (50명 기준, 추정)
| 역할 패밀리 | 주 소싱 | 내부:외부(추정) | 근거 |
|------------|--------|:---:|------|
| 컨설턴트 | 내부 전환 | 70:30 | 운영·프로세스 경험, 재교육 |
| 업종 전문가 | 내부 전환 | 80:20 | 금융·제조·공공 운영 도메인은 사내 강점 |
| 아키텍트 | 혼합 | 50:50 | 비종속 통합 설계는 외부 시드 필요 |
| AI/Data Engineer | 외부 채용 | 30:70 | AIOps/LLMOps 신기술 즉시 전력 |
| Squad Lead(L3~L4) | 외부+발탁 | 50:50 | 즉시 리더십 + 사내 핵심 발탁 |

> **운영시장 강점:** 컨설턴트·업종 전문가를 내부 운영인력 재교육으로 채울 수 있다 — 이미 고객 운영을 아는 인력이 사내에 있다는 것이 ITO/MSP 사업자의 구조적 이점이다. 전사 평균은 **"내부 65±5 : 외부 35±5"**로 통일(부록 A1), 확정값은 P1 HR 전환 풀 스크리닝으로 정한다.

### 4.7 운영 모델 — Squad 배정·상주·온콜
- **다고객 동시지원 = Squad 6개의 병렬 배정**. 1 Squad = 1 고객·1 과제를 6~8주 유지 후 다음 고객으로 재배정. 재배정 시 CoE 재사용 운영 Asset을 즉시 투입해 6~8주 안에 끝낸다.
- **상주 모델**: 상주형(금융·공공) / 하이브리드(제조·중견, 기본 권고) / 원격+CoE 백업(Asset 성숙 고객).
- **24x7 온콜**: 운영 전환 과제의 야간·주말 장애는 공통 온콜 풀(Squad 순환 + CoE SRE 백업)로 다고객 동시 커버. **AIOps로 야간 호출 자체를 줄이는 것이 1차 목표**(상주 당직 의존 탈피).
- **KPI 일원화**: 과제별·고객별 운영 대시보드(MTTR·자동화율·운영비·SLA)를 고객 보고 + 내부 평가 + 성과 정산 공용.

### 4.8 거버넌스 — 3계층 + 2개 과제 게이트
| 계층 | 회의체 | 주기 | 핵심 의사결정 |
|------|--------|------|--------------|
| 1. 과제 운영 | Squad 스탠드업/주간 과제리뷰 | 일/주 | 과제 내 우선순위·타임박스 관리 |
| 2. 사업·품질 | 월간 스티어링 | 월 | **과제 착수·전환 게이트 승인**, Squad 자원배분, 품질·리스크 |
| 3. 임원 | 분기 사업위원회 | 분기 | 손익·투자·Squad 증설/중단, 과금모델 전환 |

- **착수 게이트(G0)**: 6~8주에 끝낼 운영 과제인가? CoE Asset으로 가속 가능한가? → 통과해야 Squad 배정.
- **전환 게이트(G1)**: 운영 성과 입증됐는가? → CoE 품질 자문, 사업총괄 승인.

## 5. 프로세스·방법론

### 5.1 전체 그림 — 6~8주 운영 과제 사이클 + 상시 트랙
```
[Engagement 영업]  E1→E2→E3 ══(G0 핸드오프)══▶
[Delivery — 6~8주 타임박스 (Squad 수행)]
  ┌──────────── 운영 과제 타임박스 6~8주 ────────────┐
  │ W1~2  D0 운영 Discovery + D1 베이스라인         │
  │ W3~6  D2 PoC(섀도 우선, 운영 무중단)            │
  │ W6~8 ▣ G1 (과제→운영 전환 판정) ← 시연함정 탈출 │
  └──────────────────┬───────────────────────────────┘
       G1 통과 │ (Squad는 다음 과제로 회전 ↺)
  ═══ 상시 트랙(과제 사이클과 병행) ═══
   D3 운영전환·안정화 → D4 내재화(자동화·자산화) → ▣ G2(확장·과금) → D5 확장 ↺
```
> **전통 운영위탁과의 차이**: 전통 모델은 "사람을 상주시켜 계속 운영"한다. FDE는 **6~8주에 G1로 운영가치를 판정**하고, 상시 트랙에서 G2를 통과하며 운영에 내재화·확장한다. 수행단위가 상주 인력 풀(맨먼스)이 아닌 **Squad(4~7명) 다수 동시 가동**이다.

### 5.2 6~8주 타임박스 캘린더 (Squad 수행, 주차는 예시·가정)
| 주차 | 단계 | DoD(완료기준) | 주관 Squad 역할 | 핵심 ④툴/Asset |
|------|------|--------------|----------------|--------------|
| W1 | D0 운영 Discovery | 운영 유스케이스 1~2개+목표 KPI 서면 합의 | 컨설턴트+업종SME+Lead | 운영 진단 템플릿 |
| W2 | D1 베이스라인 | 고객이 운영 베이스라인·측정식 **서명** | AI/Data Eng+컨설턴트 | **베이스라인 백필 액셀러레이터** |
| W3~6 | D2 PoC(섀도) | G1 정량근거 데이터 확보 | 아키텍트+AI/Data Eng+SME | LLMOps 실험·룰셋·RAG·코드진단 템플릿 |
| **W6~8** | **▣ G1 게이트** | **5차원 판정 완료(통과/보류/중단)** | Squad Lead 주재 + 고객·영업·운영 | KPI·정산 템플릿 |
| (상시) | D3~D5 | 상시 트랙 각 DoD | 아키텍트·운영Lead·④툴팀 | 런북·Asset 카탈로그 |

- **타임박스 슬립 방지**: 8주차에는 반드시 G1을 판정한다(시간 기준 강제 종료).
- **상시 트랙 분리**: D3 이후(운영전환·내재화·확장)는 시간이 걸리므로 6~8주 박스 밖으로 빼 다음 과제와 병행 → Squad는 D0로 곧장 회전.

> **운영 무중단 원칙:** 운영시장 과제는 살아있는 시스템 위에서 한다. 따라서 D2는 **섀도(shadow, 실제 적용 없이 병행 관찰) 우선**으로 진행해 운영 안정성을 해치지 않는다.

### 5.3 게이트 — 시연함정 탈출 (G1 6~8주차 강제 판정)
> *시연함정 = 시연(PoC)만 반복하고 실제 운영으로 전환되지 않는 함정. 운영시장에서는 "데모는 좋았는데 운영엔 못 쓰는" 결과로 나타난다.*

| 차원 | 통과 임계치(예시·가정) | 취지 |
|------|----------------------|------|
| ① 성과 | MTTR 20%↓ **또는** 티켓 자동분류율 30%↑ | 운영 가치 실재 확인 |
| ② 정확도·신뢰 | 정확도 ≥ 합의치, 오탐률 ≤ 임계치 | 오탐 많으면 운영 부하↑ |
| ③ 운영 안정성 | 섀도/카나리 중 중대장애·SLA 영향 **0건** | 운영 안정성 훼손 금지 |
| ④ 경제성 | ROI 양(+) 또는 Payback Period(회수기간) 합의 | 운영에서 적자면 중단 |
| ⑤ 운영 수용성 | 운영팀 "운영 가능" 확인, 런북 정비 | 운영팀이 못 돌리면 실패 |
| **판정** | ①②③ 필수 + ④⑤ 합의 → 통과/보류/중단 | — |

**G1 시간 기준 룰 (타임박스 강제)**
- 6~8주차 강제 판정("측정 미완" 무기한 연장 금지).
- 보류는 단 1회·최대 +2주(누적 10주 절대 초과 금지). 보완 후에도 미달 또는 2회 연속 미달 시 **유스케이스 폐기·재정의**, Squad는 즉시 다음 과제 회전.
- 버티컬 조정: 금융·공공은 8주(안정성·감사추적), 제조는 MTTR 비중↑, 단순 과제는 6주.

**G2(상시 트랙)**: 운영 후 N개월(예 3개월) 연속 운영 KPI 유지 + 자동화 내재화 + 고객 만족 → 성과과금 전환 협상.

### 5.4 영업-운영 연계 & 품질
- **C-1 연계**: G0 핸드오프 체크리스트(6~8주 적합성·Squad 가용성 포함)·합동 착수회의, **Squad 배정·대기열 PMO**, 영업의 게이트 기준·6~8주 한계 사전 학습.
- **C-2 SLA**: AI 적용이 SLA를 개선하되 훼손 금지(악화 시 즉시 롤백). AI 자체에도 SLA 부여.
- **C-3 장애대응**: 섀도 → 사람승인 → 조건부자동의 안전 상향(자동화 수준을 단계적으로 높임).
- **C-4 변경관리(규제 핵심)**: 모델/프롬프트/런북 변경을 ITSM 변경통제에 등록·승인, 버전·감사로그.
- **C-5 모델·데이터 품질**: 정확도·오탐·드리프트 상시 대시보드, GenAI 환각·민감정보 마스킹.

### 5.5 성과 정산식
```
성과보수 = Σ[(KPI_baseline − KPI_current) × 단위가치 × 귀속계수] × 성과배분율
          (단, floor ≤ 성과보수 ≤ cap)
```
- **분쟁방지(계약 사전 정의)**: D1 서명 베이스라인 고정, 월 측정·분기 정산, 귀속계수로 외부요인 분리(운영 개선이 우리 기여분인지 분리), floor(공급사 최소수익)·cap(고객 과지급 방지). *식·수치는 예시(가정).*

## 6. 툴·플랫폼

### 6.1 한 줄 원칙 — "고객 운영환경에 얹는다(adapt), 종속시키지 않는다(no lock-in)"
- 자사 단일 플랫폼을 강요하지 않고 고객의 멀티벤더 운영툴(기존 ITSM·모니터링 등) 위에 AI를 적용한다. 운영시장 고객은 이미 운영 도구를 쓰고 있으므로, 그것을 갈아엎는 것이 아니라 그 위에 얹는 것이 핵심이다.

### 6.2 기술 스택 4레이어 (상용+오픈소스 조합)
| 레이어 | 역할 | 상용 예시 | 오픈소스 예시 | 비종속 구현 키 |
|--------|------|----------|--------------|---------------|
| L4 운영 자동화 | 티켓·런북·셀프힐링 | ServiceNow, PagerDuty | StackStorm, Rundeck, Ansible, n8n | 고객 ITSM·자동화 도구에 API/웹훅 연동 |
| L3 GenAI Ops/LLMOps | 장애보조·런북·운영에이전트·**코드진단·로그분석·RAG** | Azure OpenAI, Bedrock, Vertex | Llama·Qwen+vLLM, LangChain, Langfuse | **모델 게이트웨이**로 클라우드↔온프레미스 교체 |
| L2 AIOps | 이상탐지·RCA·예측 | Dynatrace, Datadog, BigPanda | Prometheus+이상탐지, PyOD, Keep | 알람·이벤트 표준 스키마로 엔진 교체 |
| L1 데이터·관측성 | 로그·메트릭·트레이스 수집 | Splunk, Datadog, Elastic | OpenTelemetry, Prometheus, Grafana | **OpenTelemetry 표준**으로 백엔드 교체 자유 |
| L0 보안 거버넌스 | 분류·접근통제·감사·마스킹 | (고객 IAM·DLP·SIEM) | OPA, Vault, Presidio | 고객 보안정책에 종속(우회 안 함) |

**비종속 2개 표준**: ① 수집은 OpenTelemetry(CNCF 관측성 표준)로 통일(백엔드 교체 자유) ② 모델은 게이트웨이로 추상화(클라우드↔온프레미스 오픈모델 코드 변경 없이 교체, 망분리 대응).

> **애플리케이션 운영(SM) 특화 도구:** L3에 **코드진단·로그분석·RAG**를 명시 추가했다. 애플리케이션 운영 혁신(3장 영역2)의 GenAI 코드 진단, 운영 결함 대응, 레거시 지식 RAG화가 이 레이어에서 구현된다.

### 6.3 핵심 무기 — 재사용 운영 Asset + Squad 셀프서비스 부트스트랩 ★
```
현장(Squad) 운영 Asset 생성 → Asset 카탈로그 등록 → CoE 심사·표준화
        ↑                                              ↓
  다음 운영고객에 재배포  ←  재사용(adapt)  ←  표준 Asset 라이브러리
```
- **6~8주 타임박스의 생존 조건**: Asset 카탈로그는 새 Squad가 과제 착수 시 레퍼런스 아키텍처·IaC·템플릿·런북을 끌어와 며칠 안에 환경을 띄우는 셀프서비스 키트다. W1~2 셋업을 압축할수록 W3~7 검증 시간이 확보된다. Asset이 빈약하면 6~8주가 깨진다.
- 50명·다수 Squad 동시 가동 시, **Asset 재사용률이 Squad 추가의 Marginal Cost를 결정** → 성과기반 과금 마진을 떠받친다.
- Asset 유형: 데이터·관측성 템플릿 / AIOps 룰셋·플레이북 / **GenAI 코드진단·로그분석·RAG 패턴(애플리케이션 운영)** / 자동화 런북 / 버티컬 레퍼런스 아키텍처 / KPI·정산 템플릿. *"20종·재사용률"은 목표/추정.*

### 6.4 규제산업 대응 — "망분리·온프레미스·데이터 주권"
- 데이터 주권: 운영데이터·로그·개인정보·**운영 중인 애플리케이션 소스코드**는 고객 경계 밖으로 비유출. 망분리 환경은 온프레미스 오픈모델로 GenAI 구성(외부 API 호출 없음) — 코드 진단·로그 분석도 폐쇄망 내에서 수행.
- 금융(전자금융 망분리)·공공(CSAP·국정원)·제조(OT/폐쇄망) 각각 온프레미스·프라이빗·에지 배포. SaaS 전용 도구 배제.
- *구체 제품 선정은 PoC 후 확정(비종속 원칙). 제품은 예시(option).*

## 7. 역량·인력양성·CDP

### 7.1 필요역량 — 역할 패밀리 × 시니어리티 매트릭스 (④ 동일 체계)
- 성과 단위는 개인이 아니라 **Squad**다. 5대 역량영역(A 운영도메인·B AI/데이터·C 컨설팅·D 엔지니어링·E 자산화)을 각 역할 패밀리의 주력 역량으로 매핑한다.

| 역할 패밀리 | 한 줄 정의 | 주력 역량영역 | 6~8주 타임박스 역할 |
|------------|-----------|--------------|---------------------|
| 업종/프로세스 컨설턴트 | 운영 문제 정의·성과 설득 | C(주)+A | **빠른 운영 Discovery 주도**, 범위·성과 정의 |
| 업종 전문가 | 고객 업종·운영을 깊이 아는 사람 | A(주) | 운영 맥락·현실성 검증·내재화 |
| 아키텍트 | 설계·운영 Asset 표준화 | D+E(주) | **재사용 운영 Asset 선택·조립** 설계, 품질 |
| AI/Data Engineer | AIOps·GenAI·**코드진단/RAG**·데이터 구현 | B(주)+D | AI 모델·파이프라인·자동화 빠른 구현 |
| (+) Squad Lead | Squad·고객·타임박스 총괄 | C(보조)+역할 전문성 | 범위·일정·성과 책임 |

> Squad = 컨설턴트(C)+업종전문가(A)+아키텍트(D·E)+AI/Data Eng(B) 조합으로 A~E를 모두 커버. 개인이 만능일 필요가 없다.

- **시니어리티**: 주니어(가이드 하 보조) → 미들(역할 1인분 독립) → 시니어(역할 리드·설계·고객 설득) → Squad Lead(시니어급 + 팀 리딩). 역량사전은 (역할×시니어리티) 칸별로 작성.

### 7.2 인력양성 — 50명, 역할별 트랙, 내부 운영인력 재교육 중심
| 출발점 | 강점 | 채울 갭 | 도착 역할 | 양성 기간(추정) |
|--------|------|--------|----------|----------------|
| 인프라 운영 엔지니어 | A.운영 | B.AI, 타임박스/Asset | AI/Data Eng 또는 업종전문가 | 약 5~6개월 |
| 애플리케이션 운영(SM) 개발자 | D.엔지니어링 | B.AI, E.자산화, GenAI 코드진단 | 아키텍트 또는 AI/Data Eng | 약 5~7개월 |
| 데이터/AI 인력 | B.AI | A.운영도메인, 타임박스/Asset | AI/Data Eng(즉시성↑) | 약 3~4개월 |
| 운영 PM/현업 컨설턴트 | A·C | B.AI 리터러시, Asset | 컨설턴트 또는 업종전문가 | 약 4~5개월 |
| 신규 시니어 채용 | 역할 전문성 | 사내 방법론·Asset·운영 컨텍스트 | 시니어/Squad Lead | 약 1.5~2개월 온보딩 |

> **운영시장 양성 강점:** 출발점이 "인프라 운영 엔지니어"·"애플리케이션 운영 개발자"·"운영 PM"이다 — **이미 고객 운영을 아는 내부 인력**을 재교육해 채운다. 이것이 SI 인력에서 출발하는 것보다 운영시장 적합성이 높은 이유다.

**커리큘럼(약 4~6개월)**: 공통1 AI 리터러시(3~4주) → **공통2 타임박스 운영혁신 방법론(2주, 빠른 운영 Discovery·Asset 카탈로그 활용)** → 역할 모듈(택1, 4~8주, 애플리케이션 운영 트랙은 GenAI 코드진단·RAG 모듈 포함) → 현장 OJT(Squad 합류 6~10주, 멘토 사인오프→역할 인증).

> **[로드맵 제약 — 양성 처리량 = 가동 Squad 수의 선행제약]** 한 Squad는 4개 역할 + Lead가 **모두** 채워져야 뜬다. **한 역할만 부족해도 Squad는 못 뜬다** → 역할 균형 충원이 핵심. 1기 양성(약 5개월)에 약 20~25명 확보 시 Phase 2에 약 3~4 Squad 가동(추정). 확산 속도는 **가장 부족한 역할(병목)의 양성 처리량**에 종속된다.

### 7.3 CDP — 멀티 래더 + 역할 간 이동
```
  주니어 → 미들 → 시니어 ──┬─→ [전문가 트랙] 역할별 Principal/Architect
                          ├─→ [Squad Lead 트랙] Squad Lead → 멀티 Squad 총괄
                          └─→ [관리 트랙] 조직장/사업 리더
```
- 세 트랙(전문가/Squad Lead/관리)은 **직급·보상 대등**(Google/MS의 IC 트랙 통설 차용) → 리텐션 핵심.
- **역할 간 이동(lattice)**: AI/Data Eng→아키텍트, 업종전문가→컨설턴트, 시니어→Squad Lead 등.
- 인증: (역할)×(시니어리티) Certified + 마이크로 뱃지(AIOps·GenAI·**애플리케이션 운영/코드진단**·업종·Asset 기여). 외부 인증(AWS/Azure/GCP Pro, CKA, ITIL 4)은 보조 증빙.

### 7.4 평가·보상 — 상주 맨먼스 탈피, Squad 성과 + 개인 역할 기여
| 계층 | 평가 축 | 지표(예시) |
|------|---------|-----------|
| **Squad 단위** | 운영 과제 성과 | MTTR 단축률·티켓 자동화율·운영비 절감, **6~8주 타임박스 준수율** |
| | 고객가치 | NPS/만족도, 계약 확장·갱신 |
| **개인(역할)** | 역할 수행 | 산출물 품질, Squad 내 기여도(동료평가) |
| | 자산화 기여 | 재사용 운영 Asset 등록·재사용 횟수 |

- 고정급(역할×시니어리티) + 변동급(Squad 성과분 + 개인 기여분). **자산화 보너스**(타 Squad 재사용 시 원작자 보상). **상주 맨먼스 KPI 폐지** — 더 적은 인원이 더 빨리 운영 성과를 낼수록 보상한다.

## 9. 통합 KPI 트리

> 구조: **사업성과(최종) ← 운영성과(가치증명) ← 선행지표(원인)**. 한 지표를 고객 보고·내부 평가·성과 정산에 공용으로 쓴다(KPI 일원화). 모든 지표는 **운영시장 지표**다.

```
[사업성과 — 경영진 최종 지표]
 ├─ AX FDE 매출 / 영업이익률
 ├─ 성과기반 계약 비중 (★ 맨먼스 탈피 정도, 목표/추정)
 └─ Squad당 분기 과제 처리량 / 인원 대비 고객·과제 수(커버리지 효율)
        ▲ (운영 가치로 증명되어)
[운영성과 — 고객 가치 증명 / 공용 지표]
 ├─ MTTR 단축률 / 티켓 자동화율 / 운영비 절감액
 ├─ SLA 준수율 / AI 기인 장애 0건 유지
 ├─ 알람 노이즈 감축률 / 자동복구율
 ├─ (애플리케이션 운영) SR/결함 처리 리드타임 단축 / 코드진단 시간 단축
 └─ 6~8주 타임박스 준수율 (★ Squad 회전 건전성)
        ▲ (이것이 쌓여서)
[선행지표 — 가장 먼저 움직이는 원인 지표]
 ├─ G1 과제→운영 전환율 (★ "시연함정" 핵심지표)
 ├─ 가동 Squad 수 / 역할 균형 충원율(병목 해소) (★ 확산 엔진)
 ├─ 양성 완료 수 / 운영인력 전환 수료율 / 투입 리드타임
 ├─ 재사용 운영 Asset 수 / 재사용률 / 온보딩 공수 절감률
 ├─ G2 통과 후 업셀 건수
 └─ 핵심인재 유지율 / 자발이직률
```

**Phase별 KPI 초점**
| Phase | 주력 KPI |
|-------|---------|
| P1 | 선행: 2 Squad 가동, 양성 1기 착수, Asset 카탈로그 v1, 버티컬 확정 |
| P2 | 운영성과: 첫 G1 전환율, MTTR↓·티켓 자동화율↑, 타임박스 준수율, 3~4 Squad |
| P3 | 사업성과: 성과기반 계약 비중↑, Squad당 처리량, 재사용률, 6 Squad |
| P4 | 마진·효율: 영업이익률, 인원 대비 과제 수, 핵심인재 유지율 |

> 정량 목표치(성과계약 비중 X%, 전환율 Y%, Squad당 회전 수 등)는 예산·Lighthouse Customer 실데이터 입력 후 확정(현재 미입력·추정).
